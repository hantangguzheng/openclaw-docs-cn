安全指南

沙箱配置、DM 配对、工具策略和最佳实践

⚠️ 重要提示：OpenClaw 连接真实的消息平台。请将入站 DM 视为不可信输入。本文介绍如何保护你的 OpenClaw 安装。

个人助手信任模型：OpenClaw 假设每个网关只有一个可信操作员边界（单一用户/个人助手）。它不是一个针对多个对抗性用户共享一个 Agent/网关的敌对多租户安全边界。如需混合信任或对抗性用户操作，请拆分信任边界——单独的网关 + 凭据，最好是单独的 OS 用户/主机。

🛡️ 报告漏洞与安全公告

如果你发现了 OpenClaw 的安全问题，请私下报告，不要提交公开 issue。项目的 GitHub 安全页面描述了如何报告以及需要包含什么（复现步骤、影响，以及如果可能的话最小化 PoC）。

查看 GitHub 安全公告了解过往公告和已修复版本。保持 OpenClaw 更新到已修复版本。

CVE 与更新

OpenClaw 会发布已知漏洞的安全公告。Node.js 22.12.0+ 是必需的（包含安全修复，如 CVE-2025-59466、CVE-2026-21636）。

验证你的设置

openclaw security audit --deep

加上 --fix 自动修复。检查项包括：DM 策略、allowFrom、exec 安全、网关绑定、工作区暴露。默认设置是安全的（配对、exec 拒绝、网关认证）。

🔑 网关认证与审计

对于任何非回环的使用，请保持网关认证已配置。自 2026.2.19 起，当 gateway.auth.mode 为 "none" 时，安全审计会报告 gateway.http.no_auth——回环环境下警告，生产环境下标记为严重级别。

基础架构与身份（2026.2.21）

网关锁和工具调用合成 ID 现在使用 SHA-256（而非 SHA-1）。所有者 ID 混淆使用配置中的专用 HMAC 密钥（ownerDisplaySecret），与网关 token 处理解耦，更易控制。

⚙️ 运行时与部署安全

Node.js 版本

OpenClaw 需要 Node.js 22.12.0 或更高版本。使用 node --version 验证。

Web 界面

控制 UI / Web 界面仅供本地使用。不要将其绑定到公共互联网——它未针对公开暴露进行加固。

Docker

官方镜像以非 root 用户运行。额外加固：

docker run --read-only --cap-drop=ALL \ -v openclaw-data:/app/data \ openclaw/openclaw:latest

📦 沙箱实战

许多用户在单独的机器或虚拟机上运行 OpenClaw（如专用 Mac Mini、访客 WiFi），这样 Agent 就无法访问他们的个人账户或凭据。给 Agent 自己的账户（Gmail、GitHub 等）而不是共享你的账户，可以降低风险——即使 Agent 被攻破，你的身份也不会泄露。

信任是赢来的：即使有了自己的设置，在习惯之前监督 Agent（如在发布内容前批准）是常见做法。

🔐 安全模型概览

OpenClaw 采用分层安全方法：

  • DM 配对 — 未知发送者需要明确批准
  • 沙箱 — 群组和不信任的会话可在隔离容器中运行
  • 工具策略 — 细粒度控制 Agent 可使用哪些工具
  • 白名单 — 控制谁可以与你的助手互动
  • 本地优先 — 所有数据保留在本地机器上

📬 默认 DM 访问（配对）

在 Telegram/WhatsApp/Signal/iMessage/Microsoft Teams/Discord/Google Chat/Slack 上：

  • DM 配对（dmPolicy="pairing"）— 未知发送者收到一个短配对码
  • 机器人不处理未配对联系人的消息
  • 使用 openclaw pairing approve 批准
  • 批准后，发送者被添加到本地白名单

开放 DM 访问

允许公开 DM（不推荐大多数用户）：设置 dmPolicy="open"，在白名单（allowFrom）中包含 "*"。

警告：仅在你了解安全含义并有适当沙箱配置的情况下使用开放 DM 策略。

敏感集成：邮箱和 X（Twitter）

邮箱（Gmail 等）：给 OpenClaw 访问你的主收件箱是一个重要的提示注入和钓鱼风险——入站邮件可能包含意在引导 Agent 的指令或链接。除非你需要邮件自动化，否则避免连接你的主账户；如果需要，使用专用账户或严格过滤器。

X（Twitter）：X 正在加强对机器人的执行力度。不要给 OpenClaw 自己的 X 账户或依赖它进行自动化 X 发布；账户限制很常见。建议使用其他渠道进行自动化。

🧪 沙箱配置

默认行为

  • 主会话：工具在主机上以完全访问权限运行（用于可信使用）
  • 非主会话：可配置为在 Docker 沙箱中运行

启用沙箱

将群聊和非主会话沙箱化：

"agents": { "defaults": { "sandbox": { "mode": "non-main" } } }

沙箱工具策略

默认允许列表：bash、process、read、write、edit、sessions_list、sessions_history、sessions_send、sessions_spawn

默认拒绝列表：browser、canvas、nodes、cron、discord、gateway

💬 群聊安全

群聊提供多层安全保护：

  • 群组白名单 — 控制机器人可访问哪些群
  • 提及门控 — 群中需要 @提及（默认）
  • 沙箱 — 群可在隔离的 Docker 容器中运行
  • 激活模式 — 控制机器人何时响应

🔐 认证与访问控制

网关认证

网关可通过以下方式保护：

  • Token 认证 — 非回环绑定时必需
  • 密码认证 — 通过 gateway.auth.mode 设置
  • 仅回环 — 默认绑定到 127.0.0.1 供本地访问

远程访问安全

对于远程网关访问：

  • SSH 隧道 — 加密隧道到本地主机
  • Tailscale — 带认证的安全 VPN
  • 网关 Token — 非回环访问时必需

在有适当认证之前，切勿将网关直接暴露到互联网。

⚠️ 技能安全与恶意技能

来自 ClawHub 和其他来源的技能可能存在安全风险。

ClawHub 与 VirusTotal 合作

OpenClaw 与 VirusTotal 合作，为 ClawHub 上的技能添加安全扫描。所有发布到 ClawHub 的技能现在都使用 VirusTotal 的威胁情报和 Code Insight（LLM 驱动的代码行为分析）进行扫描。

  • 哈希 + 上传 — 技能包获得 SHA-256 指纹，并在 VirusTotal 进行检查或上传分析
  • Code Insight — VirusTotal 分析 SKILL.md 和引用脚本的安全相关行为
  • 自动批准 / 阻止 — "良性"裁决的技能自动批准；可疑的标记警告；恶意技能被阻止下载
  • 每日重新扫描 — 活跃技能每日重新扫描，检测新出现的恶意更新
⚠️ 恶意技能警告：安装前务必审查技能代码，尤其是来自 ClawHub 或未知来源的。恶意技能可以：
  • 窃取加密货币钱包信息
  • 访问敏感文件和凭据
  • 执行未授权命令
  • 将数据泄露到外部服务器
  • 危害系统安全

技能安全最佳实践

  • 安装前审查——始终阅读技能代码
  • 验证来源——检查技能作者、评分和社区评价
  • 使用沙箱——在沙箱环境中运行不受信任的技能
  • 监控活动——安装新技能后观察可疑行为
  • 限制权限——使用工具策略限制技能可访问的内容
  • 定期审计——定期审查已安装的技能
  • 报告恶意技能——向 OpenClaw 社区报告可疑技能

安全策略文件（SHIELD.md）

你可以在名为 SHIELD.md 的文件中给 Agent 一个书面安全策略——社区提案，告知 Agent 如何在匹配已知威胁时做出反应（例如阻止此技能、在该工具调用前询问）。记录下来、要求批准或阻止。没有运行时强制执行；模型必须被指示遵循它，所以也继续使用沙箱和工具策略。

🔒 数据隐私

OpenClaw 的设计以隐私为核心：

  • 本地存储 — 所有数据存储在你的机器上
  • 无云同步 — 你的对话永不离开你的设备
  • 加密凭据 — 渠道凭据安全存储
  • LLM 提供商 — 只有你选择的 LLM 提供商接收消息（Anthropic、OpenAI 等）

你的工作区、记忆和配置以文件形式存储，你可以随时检查、编辑或删除。

✅ 推荐安全配置

  • 使用 DM 配对——保持所有渠道的默认配对策略
  • 启用沙箱——为群组沙箱化非主会话
  • 配置白名单——明确允许受信任的联系人
  • 使用提及门控——群聊中需要 @提及
  • 审查技能——安装前始终审查技能代码
  • 运行诊断——定期运行 openclaw doctor 检查配置问题
  • 审查日志——监控网关日志的可疑活动
  • 保持更新——定期更新 OpenClaw 获取安全补丁
  • 隔离部署——生产环境使用虚拟机、Docker 容器或专用硬件

🔍 运行安全检查

openclaw doctor openclaw security audit --deep --fix

这些检查帮助你发现：

  • 有风险的 DM 策略
  • 配置错误的沙箱设置
  • 缺失的认证
  • 安全警告
